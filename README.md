# EDIS WordPress Evidence Exporter

Version `3.7.11`, generated by EDIS Enterprise Deterministic Build Platform `3.7.11`.

Version 3.7.11 is a contract-preserving validation-hardening release. It adds source-only validation evidence tooling, explicit external-gate states, and fail-closed intake for controlled real Elementor fixtures. It does not add a user-facing feature or change installed export responsibilities. Bundle Schema `3.3.0`, Shared Envelope `2.0.0`, Package Manifest `2.1.0`, Selection Snapshot `1.2.0`, EDIS-CJ-2 and EDIS-ZIP-1 remain unchanged.

EDIS exports saved WordPress and Elementor evidence for the Python Deterministic Engine. It does not evaluate UX quality, infer browser geometry, resolve final effective values, run formulas, or call an AI service.

## 3.7.11 validation and release behavior

A successful forced private-storage self-test writes an integrity-protected, environment-bound attestation with a bounded lifetime. Ordinary application bootstrap may reuse that proof after fast path/security checks; explicit diagnostics and preflight force a live independent-process probe. `UNAVAILABLE` and `FAIL` remain blockers in every environment.

The Admin UI transfers a signed, owner-bound, short-lived preflight proof to job creation. The server verifies the normalized request, source raw hashes, owner, expiry and current storage attestation before accepting it. If any source or request field changes, full preflight must run again.

Committed artifacts and immutable input snapshots are cached only within the current PHP request. Resume boundaries still verify file hashes from disk, so a cached object cannot hide external tampering. Package assembly processes artifacts once in topological order and writes EDIS-ZIP-1 bytes directly to a temporary file before atomic commit.

Document discovery applies `edit_post` authorization before an item contributes to `items`, `total` or `total_pages`; an unauthorized `include` selection fails closed to an empty result instead of falling back to an unrestricted listing. JobStore parse caching is keyed by the SHA-256 of the bytes actually read, so a same-size/same-timestamp external rewrite is visible. Cron recovery repairs stale leases and discovers runnable jobs in one deterministic directory pass.


The source package includes `tools/validation/run-local-validation.php` and a machine-readable validation plan. The runner reports only gates that were actually executed. Composer, WordPress, Elementor, Windows/LocalWP and Python-consumer gates remain unresolved until valid external evidence is supplied. Validation files are excluded from the WordPress installation ZIP.

## Export execution

The bounded REST worker captures selected saved `_elementor_data` bytes into a protected immutable input snapshot before collectors execute. A second read detects source drift during capture. The worker then reads selected document source only from the committed snapshot.

Every completed step records:

```text
job_format_version
input_snapshot_format_version
input_snapshot_sha256
component_schema_version
implementation_version
step_input_sha256
artifact_file_sha256
```

Resume is permitted only when the job and snapshot formats are current, completed components are the exact execution-plan prefix, and all input/artifact hashes still match. In 3.7.9 the Resume transition and subsequent worker advancement occur while one stable per-job lock is held; failure to obtain that lock does not mutate the Job. Jobs created by another implementation version are not silently resumed.

WP-Cron is a recovery mechanism. The authenticated REST worker remains the primary executor. REST operations require `edis_export_evidence`, job ownership and object-level `edit_post` authorization for document operations.

## EDIS-CJ-2 lossless JSON

Version 3.7.0 introduced the frozen EDIS-CJ-2 profile; version 3.7.1 preserves it unchanged:

- valid UTF-8 is required;
- object keys are sorted by exact UTF-8 bytes;
- Unicode normalization is not performed;
- `{}` and `[]` remain distinct;
- numeric-looking object keys remain object keys;
- duplicate object keys are rejected before native PHP conversion;
- source numbers are canonicalized as exact decimal lexical nodes;
- normalization budgets reject hostile exponents and oversized expansions;
- native PHP integers remain limited to the interoperable safe range;
- non-finite native floats are rejected.

The shared vectors are in `contracts/edis-cj-2-vectors.json`.

## Dual artifact hashes

Every artifact envelope contains:

```text
semantic_payload_sha256
artifact_instance_sha256
```

The semantic hash excludes versioned operational identity only at the contract-defined envelope roots. Nested saved-source and addon properties retain their original meaning even when their names resemble operational keys such as `created_at` or `token`. The instance hash covers the complete emitted envelope except its self-referential instance-hash field. File SHA-256 values remain separate and cover emitted bytes.

## EDIS-ZIP-1

Final bundles use a pure-PHP deterministic ZIP writer rather than environment-dependent compression defaults:

- method `STORE` (no compression);
- UTF-8 byte-sorted entry paths;
- fixed DOS timestamp;
- fixed Unix file mode;
- no comments or extra fields;
- no ZIP64;
- fixed limits with fail-closed diagnostics.

This trades package size for reproducible bytes across supported runtimes.

## Durable private storage

Critical writes use exclusive temporary-file creation, complete write loops, `fflush`, `fsync`, permission application, atomic replacement/rename, parent-directory synchronization on supported POSIX runtimes, and final SHA-256 verification. The active runtime self-test also launches a separate PHP process against the exact private-storage lock file and requires it to be blocked by the held lock. If process execution is unavailable, the filesystem does not provide exclusion, the path is inside the web root, or any ancestor redirects through a symlink/junction-like path, export creation is blocked while diagnostics remain available. Per-job `.lock` sentinels are not unlinked during normal cleanup because an already-open handle must retain a stable pathname/inode identity.

### LocalWP path discovery

Local documents the WordPress document root as `<site>/app/public`. In `WP_ENVIRONMENT_TYPE=local`, EDIS 3.7.9 prefers `<site>/edis-private-storage`, retains `<site>/app/edis-private-storage` as a safe compatibility fallback, and never writes evidence under `app/public`. A storage failure activates degraded mode instead of aborting plugin activation. Use Local’s **Open Site Shell** and run `wp edis storage paths` followed by `wp edis storage self-test`.

## Privacy modes

- **Strict:** excludes original saved document objects.
- **Standard:** balanced source evidence for controlled local analysis.
- **Diagnostic:** includes additional operational metadata and requires explicit authorization confirmation.

Credentials, cookies, form values, nonces, bearer tokens and unrestricted page text are not intentionally exported.


## WordPress runtime hardening in 3.7.1

Version 3.7.1 does not change Bundle Schema `3.3.0`, Shared Envelope `2.0.0`, Selection Snapshot `1.2.0`, EDIS-CJ-2 or EDIS-ZIP-1. It hardens the WordPress integration boundary:

- conditional boot avoids loading the export application on ordinary frontend requests;
- activation/deactivation are network-aware and network activation rolls back changed sites if one site fails preflight;
- newly created multisite sites receive capability, option, storage and cleanup initialization when the plugin is network-active;
- each multisite site receives a separate private-storage namespace;
- WordPress Site Health exposes deterministic runtime, recovery schedule and asynchronous storage-integrity checks;
- WordPress privacy exporters/erasers expose and remove owner-bound operational job records;
- deleting a WordPress document removes jobs and private artifacts whose immutable selection includes that document;
- operational user meta is site-scoped on multisite;
- WordPress Filesystem transport detection is reported without allowing interactive credentials to weaken the deterministic direct-storage contract;
- WP-CLI provides status, worker, stale-job repair and storage self-test commands;
- private Job Format `2.1.0` adds explicit worker leases and heartbeat-bound recovery;
- uninstall behavior is controlled by the documented retention option and remains separate from deactivation;
- the WordPress-facing layer is subject to full WordPress-Core, WordPress-Extra and WordPress-Docs checks with documented naming exceptions for the PSR-4 core.

WP-Cron is only a wake-up/recovery trigger. Correctness remains owned by the versioned job state machine, leases, hashes and idempotent step records.

## Compatibility boundary

The WordPress layer still exports saved source evidence and deterministic indexes only. EDIS-CJ-2, dual hashing, durable storage and deterministic packaging do not move reference resolution, breakpoint resolution, class precedence, formulas, rules, runtime correlation or TruthReport generation into WordPress.


Package manifests define an explicit `semantic_identity` from the source-export root and versioned packaging policy; the per-run file inventory remains instance evidence and does not pollute semantic identity.
